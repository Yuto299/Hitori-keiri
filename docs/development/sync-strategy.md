# 同期戦略(ローカル ↔ Supabase)

> 関連: [tech-stack.md](./tech-stack.md) / 要件 [第7章 7.3](../requirements/07-non-functional.md)
> 最終更新: 2026-05-26 / ステータス: 設計(段階的に実装)

---

## 1. 方針

オフラインでも撮影・閲覧できる体験(FR-第7章 7.3)を保つため、**ローカルを正(source of truth)**としてサーバへ反映する楽観更新型を採る。

```
            書き込み                  読み込み
  UI ─────────────────▶  ローカルDB  ─────────▶ UI
                              │
                              │ 非同期で push
                              ▼
                          Supabase (Postgres)
                              │
                              │ 起動/サインイン時に pull
                              ▼
                          ローカルへマージ
```

- **書き込み**: ローカルに保存 → 即UI更新 → バックグラウンドで Supabase に push。ネット不通なら「未同期」フラグを立てて再試行(オフラインキューはフェーズ8)。
- **読み込み**: UI はローカルDBから読む(高速・オフラインOK)。起動/サインイン時に Supabase から pull し、差分をローカルにマージ。
- **競合解決**: `updated_at` の新しい側を採用(last-write-wins)。MVP は単一ユーザー=競合は稀。

## 2. 段階的な実装

| 段階 | 内容 | 状態 |
|---|---|---|
| 2-1 | ローカルのみ動作(SQLite / Web インメモリ) | 完了 |
| 2-2 | 認証(Supabase Auth・メール)導入 | フェーズ3で実装中 |
| 2-3 | サインイン時に Supabase から pull・マージ | 次の段階 |
| 2-4 | 書き込み時に Supabase へ push(失敗時リトライ) | 次の段階 |
| 2-5 | オフラインキュー(撮影時にネット不通でもキュー保存) | フェーズ8 |
| 2-6 | 画像 Storage 同期 + 保存ポリシー(FR-12) | フェーズ4以降 |

## 3. ID戦略

- ローカル ID は `r_<時刻>_<乱数>`(端末ローカルで一意)。
- Supabase 側は `uuid`(`gen_random_uuid()`)。
- **段階 2-4** で同期する際:
  - サーバへの新規 push は **ローカル ID をそのまま使わず、サーバ側で uuid を発番**
  - ローカル行に `remote_id`(uuid)を追加して紐付ける
  - 既に remote_id があれば update、無ければ insert
- これにより「複数端末で同じローカルIDを発番してぶつかる」事故を避ける(MVPでは1端末1ユーザー想定なので影響は薄いが、将来のため)。

## 4. 何を同期するか / しないか

| データ | 同期 | 備考 |
|---|---|---|
| receipts | 同期(両方向) | 中核 |
| subscriptions | サーバが正 | フェーズ6 課金で push、クライアントは pull のみ |
| category_learning | 同期(Pro機能) | 軽量。週1程度の同期で十分 |
| 画像 | サーバ単方向(端末 → Storage) | 保存ポリシーはサーバが守る |
| OCR raw | 任意 | プライバシー上、ローカルだけに残す案もあり |

## 5. RLS による安全境界

- すべてのテーブルで `user_id = auth.uid()` のRLSを必須(マイグレーション参照)。
- クライアントはサーバを信用しすぎず、サーバはクライアントを信用しない。
- 枚数カウント(FR-22)は**サーバ側で集計**して上限判定(改ざん防止)。
  - 現状は受け入れの簡潔さを優先しクライアントで集計しているが、フェーズ6課金導入時にサーバ集計へ寄せる。

## 6. 失敗時の挙動(UX)

- push 失敗: UI はそのまま、未同期マークを付与。サインアウト/ネット復帰時に再試行。
- pull 失敗: ローカルキャッシュをそのまま見せる。「サーバから取得できません」と控えめにバナー。
- 認証エラー: 自動サインアウトし、サインイン画面へ。
