## 6.1 本章の方針

レシート画像から構造化データを得る**OCR・抽出**(FR-04/05)と、店名→科目の**AI学習**(FR-06)の技術方針を定義する。第1章で「Claude API vs Google Vision を第8章で試算」としていたが、[OCRは Claude API で確定](../development/tech-stack.md#23-claude-apiocr構造化抽出)。本章はその実装方針とコスト前提を示す。

## 6.2 OCRパイプライン全体

```
[アプリ] 撮影(expo-camera) → 画像を前処理(リサイズ/トリミング)
   │  HTTPS でアップロード
   ▼
[Supabase Edge Function: ocr-receipt]   ← Claude APIキーはここ(サーバ側のみ)
   │  Claude API: Vision + Structured Outputs
   │  画像→ {date, amount_yen, store, category_candidates} のJSON
   ▼
[アプリ] 確認画面 S-03 で承認/修正(FR-07/08)
   │
   ▼
[保存] Postgres + ローカルSQLite。プランに応じ画像を保持/削除(FR-12)
```

- **APIキーをアプリに埋め込まない**。Claude呼び出しは必ず Edge Function 経由。これにより鍵保護・レート制御・枚数カウント(FR-22)の信頼境界をサーバ側に置く。
- 枚数カウントは Edge Function で集計し、クライアント改ざんを防ぐ。

## 6.3 Claude API の使い方

### 6.3.1 入力(画像)
- Claude Vision は最大 8000×8000px、base64 または Files API で送信。**画像はテキスト(プロンプト)より前**に置くのが公式推奨。
- 画像トークンは概算 `幅 × 高さ / 750`。レシート1枚は長辺を**1568px相当に収まるようリサイズ**すれば約1,500トークン程度に収まる。**送信前にクライアントでリサイズ/トリミング**してコストと精度(文字の可読性)を両立する。

### 6.3.2 出力(構造化)
- **Structured Outputs**(beta header `structured-outputs-2025-11-13`、Sonnet 4.5 / Opus 4.1 以降)でスキーマ準拠のJSONを強制する。抽出スキーマ(案):

```json
{
  "date": "YYYY-MM-DD | null",
  "amount_yen": "integer | null",
  "store": "string | null",
  "category_candidates": ["category_id", "..."],
  "confidence": { "date": 0.0, "amount": 0.0, "store": 0.0 }
}
```

- `category_candidates` は[勘定科目マスタ](./05-data-model-csv.md#535-category勘定科目マスタ)のIDから、確度順に最大3件。
- `confidence` を持たせ、低確度の項目は確認画面 S-03 でハイライトして修正を促す(誤承認の抑止)。

### 6.3.3 モデル選定
- 精度重視なら上位モデル、コスト重視なら下位モデル。**レシート抽出はコスト感度が高い**(Free無料枠の損益、[第8章](./08-financials.md))ため、まず**コスト効率の良いモデルで精度検証**し、不足分を上位モデルで補う段階的選定とする。最終選定は第8章のコスト試算と精度実測で確定。

## 6.4 抽出項目と正規化

| 項目 | 抽出方針 | 正規化 |
|---|---|---|
| 日付 | レシート記載の利用日 | `YYYY-MM-DD`。和暦・区切り違いを吸収 |
| 金額 | 税込合計を最優先 | 整数(円)。「合計」「お預り」等の誤認を回避するプロンプト |
| 店名 | 加盟店名/屋号 | 表記ゆれは AI学習のキー生成(6.6)で正規化 |
| 科目候補 | 店名・品目から推定 | マスタのIDに写像。該当なしは `misc`(雑費) |

- **金額の取り違え**(お預り・お釣り・小計を合計と誤る)が最大の事故。プロンプトで「税込の支払合計」を明示し、`confidence.amount` が低ければ確認画面で強調。

## 6.5 失敗・例外時のフロー
- **OCR失敗/低確度**:抽出できた項目だけ埋め、残りは空欄で確認画面へ。ユーザーが手入力(FR-08)できる。
- **画像が不鮮明**:再撮影を促す。
- **枚数カウント**:OCRを実行した撮影をカウント対象とする([第4章 決定事項](./04-screen-design.md#章末決定事項第4章))。再撮影・失敗で二重カウントしないよう、Edge Function 側で1撮影=1カウントに正規化する。
- **API障害**:一時保存し、リトライ可能にする(オフライン耐性は第7章)。

## 6.6 AI学習(店名→科目 / FR-06・Pro)

UIラベルは「AIが学習」だが、**実装はルールベース**([第1章 1.7.4](./01-product-overview.md#174-カテゴリai学習の実装方針参考))。

- 店名を正規化して `store_key` を作る(全角/半角・空白・記号・支店名の揺れを吸収)。
- ユーザー × `store_key` × `category_id` の**採用回数**を記録([CategoryLearning](./05-data-model-csv.md#536-categorylearningpro--fr-06))。
- 次回、同じ `store_key` のレシートは**過去採用率が最も高い科目を第一候補**に。
- ML推論・類似店名クラスタリングは Phase2。MVPは「同一店名の単純多数決」で十分「賢い」体験を出す。

## 6.7 プライバシーと画像の扱い(FR-12 との整合)
- Free:OCR完了後ただちに画像を削除。ただし抽出済みテキスト([ocr_raw](./05-data-model-csv.md#533-receipt中核) 含む)は保持しうる。
- Claude API への送信データの保持・学習利用の有無は、API のデータ利用ポリシーに従い、ユーザーに明示する(詳細は第7章)。
- レシートは個人の消費行動を含む。**最小限の項目のみ抽出・保持**し、不要な生データは持たない方針を基本とする。

## 6.8 コスト前提(第8章へ接続)
- レシート1枚 ≈ 画像約1,500トークン + 出力数百トークン。プラン別の月間枚数([第1章 1.10](./01-product-overview.md#110-収支シミュレーションの前提詳細は第8章))に掛けて損益を試算する。
- リサイズによる入力トークン削減、プロンプトの簡潔化、バッチ処理の可否が主なコスト最適化レバー。具体額は第8章。
