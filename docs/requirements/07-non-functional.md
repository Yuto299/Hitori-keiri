# 第7章 非機能要件(性能・セキュリティ・プライバシー)

> 親ドキュメント: [要件定義書インデックス](./README.md)
> ステータス: **ドラフト**(章末に確認事項あり)

---

## 7.1 本章の方針

機能要件([第3章](./03-functional-requirements.md))以外の品質特性を定義する。**個人開発1名・1ヶ月MVP**の制約下で「最低限ここは守る」ラインを明示する。過剰な作り込みはしない。

## 7.2 性能・レスポンス

| 項目 | 目標(MVP) | 根拠 |
|---|---|---|
| 撮影→確認画面の表示 | OCR込みで体感3〜5秒以内 | 「3タップ」の体験([1.4](./01-product-overview.md#14-ソリューション))を損なわない |
| アプリ起動(コールド) | 3秒以内 | ストア評価に直結 |
| 一覧スクロール | 数百件でも滑らか | Pro多枚数ユーザー(F/B)を想定。仮想化リスト |
| OCR処理 | サーバ側で非同期、待ち時間を可視化 | 失敗時リトライ可 |

- OCRはネットワーク依存のため、**処理中の状態表示**(プログレス)と**バックグラウンド継続**で体感を補う。
- 画像は送信前にクライアントでリサイズ([6.3.1](./06-ocr-ai.md#631-入力画像))し、転送量とOCRコストを抑える。

## 7.3 オフライン・同期

- **撮影・閲覧はオフラインでも可能**にする(ローカルSQLite)。OCRはオンライン必須(Claude API)。
- オフライン中の撮影は**キューに積み、オンライン復帰時にOCR・同期**する。
- ローカル(SQLite)↔ リモート(Supabase)は、**ローカルを正としつつサーバへ反映**する楽観更新。競合は「最後の更新優先(updated_at)」を基本(MVPは単一ユーザー=端末間競合は稀)。
- 機種変更・再インストール時はサーバから復元(FR-25)。

## 7.4 セキュリティ

### 7.4.1 認証・認可
- 認証は Supabase Auth(Apple/Google/メール)。トークンは `expo-secure-store` に保存(平文保存しない)。
- データ分離は **Supabase の行レベルセキュリティ(RLS)** で担保。各テーブルに「`user_id = auth.uid()` の行だけアクセス可」のポリシーを必須とする。
- **APIキー(Claude)はクライアントに置かない**。Edge Function のサーバ環境変数で保持([6.2](./06-ocr-ai.md#62-ocrパイプライン全体))。

### 7.4.2 通信・保存
- 通信は全て HTTPS。
- 端末内の機微データ(トークン)は SecureStore。レシートデータ自体は SQLite(必要なら暗号化を検討)。
- Supabase Storage の画像は**署名付きURL**で都度発行し、公開バケットにしない。

### 7.4.3 課金の信頼境界
- プラン状態・枚数カウントは**サーバ側を正**とする(クライアントの改ざんで上限回避できない)。
- レシート購入(サブスク)の検証はストアのレシート検証に従う(RevenueCat等を使う場合はその仕組み)。

## 7.5 プライバシー(レシート=個人の消費データ)

- レシートは個人の行動・嗜好を含む機微情報。**収集は目的(経費記録・CSV化)に必要な範囲に限定**する。
- **Claude API への画像送信**を、プライバシーポリシー・アプリ内で明示する。送信先・利用目的・保持の有無を説明([6.7](./06-ocr-ai.md#67-プライバシーと画像の扱いfr-12-との整合))。
- **Freeの画像即削除**は、ストレージ削減だけでなくプライバシー面でも利点(画像を持たない)。この点もユーザーに訴求できる。
- データ削除要求(退会時のデータ削除)に応じられる導線を用意する。
- App Store / Google Play の**プライバシー開示(ラベル/データセーフティ)**に正確に申告する。

## 7.6 法令・ストア要件

- **電子帳簿保存法**:MVPは非対応。ユーザー自身による原本保管が前提である旨をアプリ内に明示([第1章 1.8](./01-product-overview.md#18-mvpスコープの境界線やらないことの明示))。
- **App Store ガイドライン**:他社ソーシャルログイン(Google)を出す場合、**Sign in with Apple の併設が必要**([第4章 4.8](./04-screen-design.md#48-ログイン手段fr-25-関連))。サブスクはアプリ内課金(IAP)で提供。
- **Google Play**:データセーフティ申告・課金ポリシー順守。
- 特定商取引法・サブスク表示(価格・更新・解約方法)を設定/ストア表記で明示。

## 7.7 可用性・運用(MVP最小)

- バックエンドは Supabase のマネージドに依存(自前サーバ運用はしない)。
- エラー監視・クラッシュレポート(Sentry 等)を**MVPから入れる**(個人開発で障害に気づける手段は必須)。
- ログに**レシートの中身や個人情報を残さない**(マスキング)。
- バックアップは Supabase の標準機能に従う。

## 7.8 アクセシビリティ・国際化(MVP範囲)

- **日本語のみ**(対象が日本の個人事業主)。多言語化はしない。
- 最低限の文字サイズ対応・コントラスト(デザインAのライトテーマ)。本格的なa11yはPhase2。
- 片手操作(ギグワーカー G)を意識し、主要操作は親指の届く下部に配置([第4章](./04-screen-design.md))。

## 7.9 品質保証(テスト方針)

- **ロジックの単体テストを優先**:CSVフォーマッタ・プラン判定・枚数カウント・店名正規化(壊れると課金/出力に直結)。
- 型チェック(`tsc --noEmit`)と lint(eslint-config-expo)をCIで回す。
- 手動E2Eは「撮る→確認→保存→出力」のコアフローを最低限カバー。

## 章末:確認事項(第7章)

第7章のドラフトは以上です。以下を確認させてください。

1. **オフライン対応の範囲**:「撮影・閲覧はオフライン可、OCRはオンライン必須」としました。MVPでオフライン撮影キューまで作るか、まずは「オンライン前提」で割り切るか。
2. **SQLite暗号化**:端末内レシートデータの暗号化は「必要なら検討」に留めました。MVPで暗号化まで required にしますか?
3. **エラー監視**:Sentry等をMVPから入れる方針にしました。コスト/手間的に問題ないですか?
4. **退会・データ削除**:データ削除導線をMVPに含めます。ストア審査でも求められるため必須と考えていますが、認識合っていますか?

問題なければ第8章(収支シミュレーション)に進みます。
