## 2.1 ペルソナ設計の方針

[第1章 1.6](./01-product-overview.md#16-ターゲットセグメント概要)で示した2層をコアとしつつ、獲得チャネル・利用文脈・課金動機の異なる層を加え、計7ペルソナで設計する。A〜Cが中核(主顧客と評判の土台)、D〜Gがそれを取り囲む拡張セグメントである。

| | ペルソナ | 一言 | 主プラン |
|---|---|---|---|
| **A** | 副業ワーカー | 副業・小規模フリーランス | Light(主顧客) |
| **B** | 本業フリーランス | 本業フリーランス初期 | Pro(主顧客) |
| **C** | 超ライト層 | 年1利用・申告要否が不安 | Free(評判の土台) |
| **D** | 税理士丸投げ層 | 記帳せず税理士に渡すだけ | Light/Pro |
| **E** | 開業直後の新人 | 確定申告が初めてで不安 | Free→Light |
| **F** | 店舗系自営業 | Excel運用・会計ソフト未導入 | Pro候補 |
| **G** | ギグワーカー | 配達・運転で経費が特殊 | Light |

各ペルソナで「いつ・どこで痛みを感じ、何をきっかけに課金するか」を明確にし、[第1章 1.7.3](./01-product-overview.md#173-課金動機コンバージョン設計)の課金壁設計と接続する。なお D〜G は中核 A〜C と地続きの層であり、訴求文言や機能の優先度を決めるための解像度として定義する(全員を同等に追うわけではない)。

## 2.2 ペルソナA:副業ワーカー「佐藤 健太」(Light主顧客)

| 属性 | 内容 |
|---|---|
| 年齢 / 職業 | 32歳 / 会社員(本業)+ Web制作の副業 |
| 年間副業売上 | 約180万円 |
| レシート枚数 | 月20〜40枚(打合せのカフェ代、書籍、ガジェット、交通費) |
| 会計リテラシー | 確定申告は3年経験あり。だが勘定科目は毎年ググる。簿記の知識はない |
| 現在の運用 | レシートを封筒に溜め、2月にExcelへ手入力。freeeを一度試したが「機能が多すぎて」3日で挫折 |
| デバイス | iPhone(メイン)。PC作業は本業中心で副業はスマホ完結したい |

**痛み(ペイン)**
- 確定申告期に40枚×12ヶ月のレシートを前に手入力で消耗する
- 経費か私用か曖昧なレシートの判断を後回しにし、結局思い出せない
- freeeは「ちゃんとした人向け」で、自分には過剰かつ割高に感じる

**ひとり経理での体験**
- 打合せ直後にその場で撮影 → メモに「同席者・案件名」を一言。後で思い出す負担が消える
- 申告期、freee形式CSVを書き出して税理士 or 会計ソフトに渡すだけ
- 月¥480なら「年1回の絶望を回避する保険」として許容できる

**課金トリガー**:申告期、汎用CSVが会計ソフトにインポートできないと気づいた瞬間に **Free → Light**。

## 2.3 ペルソナB:本業フリーランス「鈴木 美咲」(Pro主顧客)

| 属性 | 内容 |
|---|---|
| 年齢 / 職業 | 38歳 / フリーランスのフォトグラファー兼デザイナー |
| 年間売上 | 約650万円 |
| レシート枚数 | 月80〜150枚(機材、ロケ交通費、外注、交際費が多い) |
| 会計リテラシー | 青色申告(65万円控除)。freee契約中だが入力が追いつかず溜める |
| 現在の運用 | freee契約済みだが「魔法スキャン」の精度と手間に不満。結局月末にまとめて格闘 |
| デバイス | iPhone + iPad。移動が多く外出先での記録ニーズが高い |

**痛み(ペイン)**
- レシート枚数が多く、月末・申告期の処理が重い
- 「あの機材どの案件で買ったっけ」と後から経費を遡って探したい場面が頻発
- 交際費の同席者記録を忘れ、税務調査時に説明できるか不安
- freeeに月¥1,000以上払っているが、使っているのは実質レシート機能だけ

**ひとり経理での体験**
- 撮影直後にPro音声入力で「クライアントBのロケ、同席〇〇さん」と片手記録
- 店名→科目のAI学習で、よく行く店は科目が自動で第一候補になる
- 無期限保存+検索で「3ヶ月前のあの機材」をすぐ呼び出せる
- freeeを解約し、ひとり経理(Pro ¥780)+ 税理士で運用をスリム化

**課金トリガー**:過去のレシートを遡れない/30日で画像が消えると気づいた瞬間に **Light → Pro**。検索・無期限保存・AI学習が決め手。

## 2.4 ペルソナC:超ライト層「田中 由美」(Free滞留/離脱層)

| 属性 | 内容 |
|---|---|
| 年齢 / 職業 | 29歳 / 会社員 + 単発の副業(年数回のイラスト受注) |
| 年間副業売上 | 約25万円(雑所得 or 少額の事業所得) |
| レシート枚数 | 月0〜5枚 |
| 会計リテラシー | 確定申告そのものが不安。やるべきか毎年迷う |
| 現在の運用 | レシートはほぼ放置。申告は「たぶん不要」と自己判断しがち |

**痛み(ペイン)**
- そもそも自分が申告すべきか分からない
- 経費管理に月額を払うほどの売上規模ではない

**ひとり経理での体験**
- Freeで月5枚まで撮影。汎用CSVで最低限の記録は残せる
- ほとんどのケースで課金には至らない

**このペルソナを定義する理由**
- **獲得コストと評判管理の観点**で重要。Freeの「画像即削除」を誤解するとレビューで「写真が消えた」と低評価を付けるリスクがある層
- [第1章 1.7](./01-product-overview.md#172-機能比較) 注1の事前明示UXは、まさにこの層の誤解防止が目的
- 課金しないが、口コミ・ストア評価を通じてペルソナA/Bの流入に影響する

## 2.5 ペルソナD:税理士丸投げ層「高橋 誠」(Light/Pro)

| 属性 | 内容 |
|---|---|
| 年齢 / 職業 | 45歳 / 個人事業のコンサルタント |
| 年間売上 | 約900万円 |
| レシート枚数 | 月40〜80枚 |
| 会計リテラシー | 簿記は分からないし覚える気もない。記帳・申告は顧問税理士に一任 |
| 現在の運用 | レシートを封筒・レジ袋に溜め、月1〜四半期に1回まとめて税理士へ郵送/持参。税理士から「もう少し整理して」と毎回言われる |
| デバイス | iPhone。本業が忙しく経理に時間を割きたくない |

**痛み(ペイン)**
- レシートを溜めて渡すだけだが、その「溜める・分類する」作業すら面倒
- 税理士に渡す前の仕分けで毎回手間取り、関係がややギスギスする
- 領収書の紛失で経費計上漏れが起きていそうだが確認しようがない

**ひとり経理での体験**
- 撮ったそばからCSV化されるので、税理士には**CSVを送るだけ**で済む
- 紙の山を仕分けする作業が消え、紛失リスクも下がる
- 自分は科目を深く理解しなくても、候補を承認するだけでよい

**課金トリガー**:税理士から「対応形式のCSVで欲しい」と言われた/紙の郵送をやめたい瞬間に **Free → Light**。枚数が多く遡り確認のニーズが出れば **Pro**。

> 補足:第1章で法人プランを「Pro拡張型」と仮置きしたが、この層が将来「税理士連携」ニーズの起点になりうる。Phase2の連携機能を検討する際の基点ペルソナ。

## 2.6 ペルソナE:開業直後の新人「伊藤 さくら」(Free→Light)

| 属性 | 内容 |
|---|---|
| 年齢 / 職業 | 27歳 / 今年フリーランスのライターとして開業 |
| 年間売上 | 約120万円(初年度、これから伸ばす想定) |
| レシート枚数 | 月10〜25枚 |
| 会計リテラシー | **確定申告が初めて**。開業届・青色申告承認申請も手探り。経費の範囲が分からない |
| 現在の運用 | とりあえずレシートは取ってあるが、何が経費になるか判断できず不安なまま放置 |
| デバイス | iPhone。SNSで「フリーランス 確定申告」を検索しがち |

**痛み(ペイン)**
- そもそも「これは経費になるのか」の判断に自信がない
- 申告期が近づくほど不安が募るが、何から手をつけるべきか分からない
- いきなり有料の会計ソフトに登録する勇気はない

**ひとり経理での体験**
- まずFreeで撮影を始め、「撮るだけでいい」体験で心理的ハードルが下がる
- 科目候補が自動提示されるため、経費判断の最初のとっかかりになる
- 申告期にCSVが必要と気づき、初めての申告を乗り切る相棒としてLightへ

**課金トリガー**:初めての申告期、CSVが必要だと気づいた瞬間に **Free → Light**。ここで良い体験をすると、売上成長に伴いA/B層へ育つ。

> 補足:**LTVの観点で最重要**になりうる層。開業初年度に獲得すれば、売上成長とともにLight→Proへ自然に成長する「育てる顧客」。オンボーディングと安心感の設計が鍵。

## 2.7 ペルソナF:店舗系自営業「渡辺 一郎」(Pro候補)

| 属性 | 内容 |
|---|---|
| 年齢 / 職業 | 52歳 / 個人経営の小さな飲食店オーナー |
| 年間売上 | 約850万円 |
| レシート枚数 | 月80〜150枚(食材仕入れ、備品、光熱費、雑費が大量) |
| 会計リテラシー | 長年Excelと電卓で自己流管理。会計ソフトは「難しそう」で未導入 |
| 現在の運用 | レシートを箱にため、月末に妻とExcel入力。仕入れ伝票が多く処理が重い |
| デバイス | Androidスマホ。PCは苦手意識がある |

**痛み(ペイン)**
- 仕入れ系レシートが大量で、Excel手入力が毎月の重労働
- 会計ソフトは高機能すぎ・難しそうで導入に踏み切れない
- 紙のレシートが箱に溢れ、見返したい時に探せない

**ひとり経理での体験**
- 仕入れレシートをその場で撮影 → 箱に溜める運用から解放される
- 汎用/対応形式CSVで、Excel運用や税理士へのデータ受け渡しが楽になる
- 無期限保存+検索(Pro)で過去の仕入れレシートを遡れる

**課金トリガー**:枚数が多く30日保存では足りない/遡り検索が欲しいと気づいた瞬間に **Pro**。Androidユーザーが一定数いる点はストア施策に影響。

> 補足:売上規模は事業所得が中心で個人事業主の範囲。**Androidの比率が高い**層であり、第1章のクロスプラットフォーム前提(Expo)の価値が効く。

| 属性 | 内容 |
|---|---|
| 年齢 / 職業 | 34歳 / フードデリバリー配達 + 軽貨物運送 |
| 年間売上 | 約350万円 |
| レシート枚数 | 月15〜40枚(ガソリン、駐車場、車両整備、通信費、備品) |
| 会計リテラシー | 確定申告は必要と理解。家事按分(車・スマホ)に毎年悩む |
| 現在の運用 | ガソリンレシートをダッシュボードに溜める。申告期にまとめて格闘 |
| デバイス | Androidスマホ。運転中・配達の合間で**片手操作**が基本 |

**痛み(ペイン)**
- 経費がガソリン・車両・通信に偏り、家事按分の扱いが毎年わからない
- 運転や配達の合間で、ゆっくり経理する時間がない
- レシートを車内に溜めて劣化・紛失しがち

**ひとり経理での体験**
- 給油直後に**その場で撮るだけ**。片手で完結する撮影体験が刺さる
- (Pro)音声入力で「〇〇向け配送のガソリン代」と運転前後に片手記録
- メモ欄に按分メモを残し、申告期の判断材料にできる

**課金トリガー**:申告期にCSVが必要と気づいた瞬間に **Free → Light**。音声入力・遡り検索が効けば **Pro**。

> 補足:**「移動が多く片手・短時間で記録したい」体験ニーズ**を代表する層。撮影フローの速さ(3タップ)とPro音声入力の価値が最も直接的に効く。Androidユーザーが多い。

## 2.9 ペルソナ間の関係と狙う順序

### 2.9.1 中核ペルソナ(A〜C)

| | A 佐藤 | B 鈴木 | C 田中 |
|---|---|---|---|
| 主プラン | Light | Pro | Free |
| 月間レシート | 20〜40枚 | 80〜150枚 | 0〜5枚 |
| 最大の痛み | 申告期の手入力地獄 | 多枚数処理+遡り検索 | 申告要否そのものが不安 |
| 課金の瞬間 | CSVがインポートできない | 画像が消える/遡れない | (基本課金しない) |
| 事業上の役割 | 売上のボリュームゾーン | ARPU最大化 | 評判・流入の土台 |

### 2.9.2 拡張ペルソナ(D〜G)

| | D 高橋 | E 伊藤 | F 渡辺 | G 中村 |
|---|---|---|---|---|
| 主プラン | Light/Pro | Free→Light | Pro候補 | Light |
| 月間レシート | 40〜80枚 | 10〜25枚 | 80〜150枚 | 15〜40枚 |
| 最大の痛み | 仕分け・郵送が面倒 | 経費判断に自信がない | 仕入れの大量入力 | 片手・短時間で記録したい |
| 課金の瞬間 | 税理士にCSVで渡したい | 初めての申告期 | 30日保存では足りない | 申告期にCSVが必要 |
| 事業上の役割 | 税理士連携の起点 | 育てる顧客(高LTV) | Android+多枚数の検証 | 撮影体験ニーズの代表 |

### 2.9.3 獲得の優先順位

1. **A(Light)を主軸に獲得 → 申告期にB(Pro)へ引き上げ**(中核の二段ロケット)
2. **E(開業直後)を初年度に獲得し、A/Bへ育てる**(LTV最大化の長期投資)
3. **D・F・Gは訴求文言と機能優先度の解像度**として活用。D=税理士へのCSV受け渡し、F=Android+多枚数、G=片手・音声入力 が、それぞれ刺さる切り口
4. **C は積極獲得対象ではない**が、Freeの設計品質(特に画像削除の事前明示)で評判を守る

> 重複の扱い:D(税理士丸投げ)・F(店舗系)は「大量処理・受け渡し」の痛みで B と一部重なるが、**獲得チャネル・利用文脈が異なるため別ペルソナとして残す**(統合しない)。Bは「freeeから乗り換える本業層」、Dは「記帳しない・税理士前提層」、Fは「会計ソフト未導入のExcel/店舗層」と、刺さる訴求が異なる。

各ペルソナの痛みは、第3章の機能要件・ユーザーストーリーで具体的な要件に落とし込む。

## 章末:確定事項(第2章)

確認事項は以下のとおり確定した(計7ペルソナ:A〜C 中核 / D〜G 拡張)。

1. **ペルソナ数** → **7ペルソナのまま維持**(A〜C 中核 + D〜G 拡張)。解像度を保つことを優先する。
2. **売上規模** → A=180万 / B=650万 / C=25万 / D=900万 / E=120万 / **F=850万(1,200万から下方修正)** / G=350万 で確定。Fは個人事業主の現実的な範囲に寄せた。
3. **重複の許容** → **重複を許容して残す**(D・FはBと一部重なるが、獲得チャネル・文脈が異なるため統合しない。上記の重複の扱いを参照)。
4. **獲得優先順位** → 2.9.3 の「A獲得 → 申告期にB引き上げ → Eを育てる」を主軸として確定。

第2章は確定。

問題なければ第3章(機能要件 / ユーザーストーリー)に進みます。
